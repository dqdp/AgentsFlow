# v0.2 Next Slices Plan

Status: Planning note

Date: 2026-06-21

## Purpose

This note parks the next major work slices after the current v0.2 readiness
cleanup. It is intentionally not an `AGENTS.md` rule and not a workflow
promotion decision. It records likely sequencing so the project does not lose
the discussion while keeping the v0.2 supported path narrow.

The current v0.2 supported application path remains:

```text
project-initialization.prepare-workflow -> big-feature-contract-first
```

## Recommended Sequence

### 1. Define PR Merge Readiness Utility

Goal: define a small workflow or utility pattern for deciding whether an
AgentsFlow branch is ready to open, accept or merge as a pull request.

Rationale: this is the nearest practical need before merging v0.2 work into the
main branch. It should validate the current branch without turning AgentsFlow
into a release-management runtime.

Expected scope:

- identify branch, base and intended commit range;
- verify clean or intentionally documented worktree state;
- run repository validation and tests;
- check source-of-truth documentation consistency;
- run a review gate over the branch evidence;
- use live external Claude review evidence when subscription-local Claude is
  available, while keeping mock smoke tests as the CI-safe baseline;
- relevance-validate P0/P1 candidate findings and use collision-control when a
  blocker-level candidate is rejected or downgraded;
- record human merge approval and a post-merge verification plan.

Implementation approach:

- implement this utility through the existing `big-feature-contract-first`
  workflow: contract first, plan/design readiness, red-capture for new
  validators/examples, implementation, green verification and review;
- keep the feature scope narrow: workflow/utility docs, templates, examples and
  validators needed for PR merge readiness, not a release-management runtime.

Accepted decisions for implementation:

- workflow / utility name: `pr-merge-readiness`;
- status: v0.2 utility workflow, similar in promotion level to
  `review-only-fusion`, not a primary application workflow;
- implementation method: use `big-feature-contract-first` to build this utility
  with a contract-first, red-capture, green-verify and review sequence;
- first slice shape: include docs, schema, template, example and validator
  coverage; do not stop at prose-only documentation;
- live Claude evidence policy: store normalized reports and invocation metadata
  as normal run artifacts; store raw output only when the packet is
  non-sensitive, otherwise store redacted raw evidence or an explicit
  summary/pointer with the reason; never present mock evidence as a live Claude
  invocation.

Candidate artifacts:

- `pr-merge-readiness-report.md` or `.json`;
- review prompt contract, review packets and invocation set when review is used;
- finding-validation or gate-triage report;
- human merge decision record;
- post-merge verification note.

Non-goals:

- no release platform;
- no package distribution workflow;
- no generic CI provider abstraction;
- no new multi-provider runtime.

Remaining decisions before implementation:

- exact field set for the readiness report and human merge decision record;
- exact validator strictness for sensitive live Claude raw output redaction.

### 2. Run PR Merge Readiness For AgentsFlow Merge

Goal: use the new `pr-merge-readiness` utility as the concrete acceptance run
for the AgentsFlow v0.2 pull request and merge to `main`.

Rationale: defining the utility is not enough. The v0.2 branch should be accepted
through the same process it introduces, so the first real run is the merge
readiness decision for AgentsFlow itself.

Expected scope:

- identify the branch, base branch and intended commit range;
- verify there are no unrelated or unexplained working-tree changes;
- run repository validation and tests;
- check README, checkpoints, ADRs, workflow model, MVP standard and provider
  docs for source-of-truth consistency;
- run the configured review gate for the PR merge readiness packet;
- include live Claude review evidence when local subscription auth is available;
- relevance-validate all P0/P1 candidate findings, including collision-control
  if any blocker-level candidate is rejected or downgraded;
- record the human decision to accept, defer or reject the PR/merge;
- after merge, run the declared post-merge verification check.

Candidate artifacts:

- PR merge readiness report;
- review packet and review prompt contract;
- review invocation set when provider-assigned review is used;
- finding validation or gate triage report;
- human merge decision record;
- post-merge verification evidence.

Exit condition:

- PR/merge is accepted only when repository checks are green, required evidence
  is present, no P0/P1 finding remains validated, and the human merge decision is
  recorded.

Non-goals:

- do not make the full Bro dogfood run a pre-merge blocker unless a separate
  human decision explicitly upgrades it to release-blocking evidence;
- do not require live Claude evidence in CI or in environments without
  subscription-local auth; record the blocker or use the CI-safe mock baseline
  according to the utility's policy.

### 3. Bro Tools Gateway TG-A Dogfood Run

Goal: rerun the Tools Gateway TG-A feature in `/Users/alex/Bro` from a fresh
branch and fresh AgentsFlow run, using the merged v0.2 methodology.

Rationale: this is the best end-to-end dogfood test for the current AgentsFlow
methodology, especially project initialization, provider/model review
assignment, external Claude review, normalized reviewer reports and review-gate
triage. By default it is post-merge validation of the accepted methodology, not a
pre-merge release blocker.

Expected scope:

- create a new Bro branch before changes;
- start a clean AgentsFlow run instead of continuing an older incomplete run;
- pin the merged AgentsFlow upstream commit in the Bro run evidence;
- run `project-initialization` with `intent_mode: prepare-workflow` and
  `target_workflow: big-feature-contract-first`;
- make the human-confirmed documentation legacy adoption decision before using
  project documentation as authority;
- prepare TG-A contract/design/readiness artifacts before implementation;
- run implementation with red-capture and green-verify framing;
- run provider-assigned review gates with external Claude launched first;
- compare the result with the earlier TG-A attempt.

Candidate comparison dimensions:

- artifact completeness;
- architecture/design clarity;
- implementation diff quality;
- review findings and false positives;
- friction in human decisions;
- gate evidence quality;
- time/cost of external review;
- places where AgentsFlow was helpful vs too heavy.

Non-goals:

- do not start this run from an old incomplete state;
- do not treat previous TG-A decisions as automatically accepted unless the
  human explicitly confirms them for the new run;
- do not change AgentsFlow upstream during the Bro run unless a methodology
  blocker is discovered.

### 4. Standalone Knowledge Extraction Workflow

Goal: promote knowledge extraction from a project documentation corpus or
knowledge base into a candidate standalone workflow.

Rationale: `project-initialization` already includes documentation legacy
adoption and can produce `project-knowledge-extraction.md`, but extracting
project knowledge from a larger corpus can be a substantial task on its own.

Expected scope:

- define an explicit source corpus and extraction purpose;
- classify source documents and authority boundaries;
- separate observed facts, human-confirmed decisions, inferred knowledge,
  historical context and unresolved questions;
- record provenance, confidence and human-confirmation needs;
- choose persistence scope: run-level or project-level;
- produce normalized extraction artifacts that can seed packs, ADR candidates,
  workflow bindings, term maps or operating decisions when approved.

Candidate artifacts:

- source corpus inventory;
- extraction policy / decision record;
- `project-knowledge-extraction.md` and possibly a structured companion JSON;
- unresolved-questions register;
- promotion plan for project-level artifacts when persistence is project-level.

Non-goals:

- no silent rewrite or deletion of source documentation;
- no automatic promotion of extracted knowledge to project authority;
- no parallel replacement for `project-documentation-disposition.yaml` or human
  documentation legacy adoption decisions.

Open decisions before implementation:

- standalone workflow name and status;
- whether this starts as a workflow, skill, or utility pattern;
- whether to add a structured schema for extracted knowledge in the first slice;
- how deep extraction should interact with project packs, ADR candidates and
  operating decisions.

## Current Recommendation

Implement the PR merge readiness utility first because it directly supports
the upcoming v0.2 merge. Then immediately run that utility as the concrete
acceptance process for the AgentsFlow v0.2 pull request and merge. Use the Bro
TG-A dogfood run after merge as practical validation of the accepted methodology.
Treat standalone knowledge extraction as a candidate workflow after the
PR merge readiness and Bro dogfood evidence clarify how much structure is
actually needed.
